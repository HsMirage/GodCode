# CodeAll 任务承接能力优化方案（2026-03-08）

> 文档日期：2026-03-08  
> 文档定位：补充 `docs/plans/2026-03-06-comprehensive-optimization-plan.md`，聚焦“当前项目给一个任务后，Agent 能否稳定完成”的能力评估与优化路线。  
> 目标读者：产品负责人、架构负责人、Agent/Workforce/Delegate 模块开发者、测试与发布负责人。

---

## 1. 文档目标

本方案不再泛泛讨论“项目功能是否存在”，而是聚焦以下三个更关键的问题：

1. **当前 CodeAll 接到一个真实任务后，能否完成？**
2. **哪些任务可以稳定完成，哪些任务只能部分完成，哪些任务当前不适合交给它？**
3. **如果要把 CodeAll 提升为“可托付的任务执行系统”，接下来最应该补什么？**

本方案的输出目标是：

- 给出一份**面向任务完成率**的现状判断
- 识别**阻碍 Agent 真正稳定交付**的关键短板
- 形成**按优先级排序的优化路线图**
- 为后续任务派发建立统一的**任务模板、验收模板和度量指标**

---

## 2. 当前结论摘要

### 2.1 一句话结论

**CodeAll 已经具备“接任务并完成一部分真实工作”的能力，但尚未达到“高风险场景下可长期无人值守自治交付”的成熟度。**

### 2.2 当前能力判断

按“给它一个任务，它能不能完成”来看，当前项目可分为三档：

| 档位 | 结论 | 适用任务类型 | 当前判断 |
|---|---|---|---|
| A | **能做** | 明确边界、局部修改、信息充分、验收简单的任务 | 可稳定承接 |
| B | **勉强能做** | 跨文件、多步骤、需拆解和重试的任务 | 依赖模型质量、任务描述质量、环境稳定性 |
| C | **目前做不好** | 高风险、高权限、跨平台发布、长时自治、强审计要求任务 | 不建议完全放手 |

### 2.3 对应任务示例

#### A 档：当前可以放心交给它的任务

- 修复一个已定位的 bug
- 改动 1~5 个文件以内的功能点
- 根据指定文件补文档/注释/测试
- 按已知命令执行指定测试并汇总结果
- 对某个模块做只读分析、说明、审查

#### B 档：当前可以交给它，但要有人盯的任务

- 跨模块重构
- 需要任务拆解与并行执行的开发任务
- 需要浏览器工具、外部网页、文件写入、shell 命令混合协作的任务
- 需要自动恢复、重试、续跑的中长链路任务

#### C 档：当前不建议完全自治的任务

- 自动修改大量文件并直接发布
- 涉及敏感 shell、危险写入、凭据操作的任务
- 需要严格“人工审批后再执行”的任务
- Windows/macOS/Linux 跨平台发包与验收闭环
- 超长时运行、跨重启、跨会话的复杂自治工作流

## 3. 现状依据（从任务执行闭环角度）

下面不是能力宣传，而是从“任务是否能闭环完成”角度看当前主链路。

### 3.1 已具备的主能力

#### 3.1.1 已有任务入口与执行路由

- 普通消息可直接进入模型执行链：`src/main/services/message/message-execution.service.ts`
- 消息可根据策略走 `direct / delegate / workforce`：`src/main/services/message/message-execution.service.ts`、`src/main/services/router/smart-router.ts`
- Workforce 能根据复杂度和指令进入工作流编排：`src/main/services/router/smart-router.ts`

这说明系统并不是“只有聊天界面”，而是真正有任务路由骨架。

#### 3.1.2 已有 Delegate 原子执行闭环

- Delegate 会先创建任务记录：`src/main/services/delegate/delegate-engine.ts`
- 然后解析模型绑定与 fallback：`src/main/services/llm/model-selection.service.ts`
- 再为当前 Agent/Category 限制工具作用域：`src/main/services/delegate/tool-allowlist.ts`
- 再执行工具调用与 LLM 循环：`src/main/services/tools/tool-execution.service.ts`
- 最后落库完成/失败状态：`src/main/services/delegate/delegate-engine.ts`

说明当前系统已经具备“原子任务接收 -> 执行 -> 记录结果”的核心能力。

#### 3.1.3 已有 Workforce 编排能力

- 工作流入口存在：`src/main/services/workforce/workforce-engine.ts`
- 支持计划文件解析与 LLM 分解两种任务来源：`src/main/services/workforce/workflow-decomposer.ts`
- 支持依赖图构建与死锁/环检测：`src/main/services/workforce/workflow-graph-builder.ts`
- 支持并发调度与限流：`src/main/services/workforce/worker-dispatcher.ts`
- 支持 observability 快照写出：`src/main/services/workforce/workflow-observability-writer.ts`

说明当前系统已经不是“伪多 Agent”，而是有真实的拆解、调度、执行和结果聚合。

#### 3.1.4 已有持久化、恢复与追踪能力

- 消息持久化：`src/main/services/message/message-persistence.service.ts`
- 会话状态恢复：`src/main/services/session-state-recovery.service.ts`
- 任务续跑与恢复：`src/main/services/task-continuation.service.ts`
- 工作流恢复分类与恢复路由：`src/main/services/workforce/workflow-recovery-controller.ts`
- UI 侧任务/Agent 状态查看：`src/renderer/src/store/agent.store.ts`

说明它有一定的“失败后继续做事”的基础，而不是一次性脚本。

### 3.2 当前能力上限的真实限制

#### 3.2.1 “需要确认”的危险工具并没有真正停下来等确认

当前权限系统已经能把工具标成 `confirm`，比如：

- `bash`
- `file_write`
- 一部分浏览器主动交互工具

但当前执行器真正阻止执行的条件是“policy 不允许”，而不是“需要用户确认”。

这意味着：

- 系统**知道**这是危险工具
- 但系统**没有真正建立审批闸门**
- 最终表现更像“带警告继续执行”，而不是“等待人工批准”

相关模块：

- `src/main/services/tools/permission-policy.ts`
- `src/main/services/tools/tool-execution.service.ts`

这是当前从“可用”走向“可托付”最大的阻塞项。

#### 3.2.2 中间执行过程的耐久化不足

当前流式输出与工具过程状态主要停留在 renderer 内存态：

- `src/renderer/src/store/streaming.store.ts`

最终 assistant 消息会落库，但中间过程如果：

- 页面刷新
- 窗口崩溃
- 进程异常退出
- 会话切换

就可能只留下最终结果，或直接丢掉中间上下文。对于长任务，这会显著影响：

- 故障恢复
- 人工接管
- 任务审计
- 二次续跑

#### 3.2.3 Router 目前主要还是规则路由

当前 `SmartRouter` 主要依赖：

- 正则规则
- 显式提示词
- 若干复杂度关键词

优点是简单、可控。缺点是当任务表达方式变化时，容易出现：

- 路由策略大体对，但分配不够精确
- 复杂任务走了错误模式
- 该分给只读 Agent 的任务被交给了执行 Agent

相关模块：`src/main/services/router/smart-router.ts`

#### 3.2.4 长链路恢复仍偏“恢复到能继续”，还没达到“恢复到充分理解”

当前恢复链路已经具备：

- crash/manual/auto/context 多来源恢复
- recovery metadata 透传
- workflow retry / recovery 分类

但上下文恢复仍偏结构性恢复，而不是语义性恢复。也就是：

- 能知道“继续做这个任务”
- 但未必能完整恢复“为什么做到这一步、还差什么、哪些决策不能推翻”

这会影响长任务、多阶段任务和大仓库任务的收敛质量。

#### 3.2.5 发布与跨平台能力尚未闭环

仓库已有正式验收记录表明：

- mac 构建与启动闭环较完整
- delegate / workforce / browser 核心链路已可验证
- 但 Windows 构建和启动仍未闭环
- 单测、集成、E2E 也未全绿

相关记录：

- `docs/final-acceptance.md`
- `docs/test-reports/release-preflight-2026-03-03.md`

这意味着：当前系统更适合做“研发辅助 Agent”，而不是“可直接拿去发布托底的自治交付平台”。

## 4. 根因分析：为什么“有很多能力”不等于“能稳定完成任务”

从当前项目状态看，核心问题并不是“功能缺失太多”，而是以下四个层面的成熟度还没完全补齐。

### 4.1 任务定义成熟度不足

Agent 系统对“边界清晰、输入充分、验收明确”的任务表现通常较好；对模糊任务则容易出现：

- 任务范围膨胀
- 路由偏差
- 多余修改
- 验收口径不一致

因此，问题并不全在 Agent，任务本身的结构化程度也直接决定完成率。

### 4.2 执行治理成熟度不足

现在有：

- 工具作用域限制
- 权限模板
- Hook 治理
- 审计与 run log

但还缺：

- 真正的审批暂停点
- 高风险动作前的显式确认状态
- 失败后的统一人工接管入口
- 任务级“允许修改范围”强约束

### 4.3 过程可观测性成熟度不足

当前系统已经有：

- workflow observability snapshot
- agent run log
- task 状态追踪

但仍缺：

- 工具调用级持久化事件流
- 任务中间态 checkpoint
- session 刷新后完整恢复执行现场
- “当前为什么卡住”的统一诊断面板

### 4.4 评估体系成熟度不足

当前仓库已有很多测试与报告，但还缺一套专门衡量 Agent 任务交付能力的指标，例如：

- 任务完成率
- 一次通过率
- 平均返工次数
- 人工接管率
- 任务越界率
- 高风险动作审批命中率

没有这些指标，就很难判断每轮优化到底有没有真正提高“任务可完成性”。

## 5. 优化原则

接下来的优化不建议继续走“哪里有问题补哪里”的散点修补方式，而应统一遵循以下原则。

### 5.1 先提升“托付感”，再提升“聪明度”

优先级应是：

1. 安全可控
2. 可追踪可恢复
3. 结果可验收
4. 路由更智能
5. 完全自治

### 5.2 先补执行闭环，再补体验细节

短期最重要的不是新 UI，而是保证：

- 危险操作不会无确认执行
- 长任务不会轻易丢上下文
- 失败之后能明确知道怎么接手
- 发布链路有真实门禁

### 5.3 先把“任务类型边界”做清楚，再追求通用万能 Agent

建议先明确：

- 哪些任务默认允许 direct
- 哪些任务默认必须 delegate
- 哪些任务默认必须 workforce
- 哪些任务默认只读
- 哪些任务必须人工审批后才能继续

### 5.4 一切优化都必须落到可量化指标

如果无法回答下面这些问题，优化就不算真正完成：

- 完成率提高了多少？
- 失败类型减少了多少？
- 人工接管是否减少？
- 危险操作是否真正被闸住？

## 6. 分阶段优化路线图

下面的路线图按优先级分为 P0 / P1 / P2 / P3。

---

## 6.1 P0：把系统从“可用”提升到“可托付”

### P0-1 真正实现工具审批闸门

#### 目标

将当前 `confirm` 权限从“标记需要确认”升级为“暂停执行，等待用户批准后再继续”。

#### 现状问题

- 权限系统能识别 `confirm`
- 执行器没有审批状态机
- 危险工具可能直接执行

#### 改造范围

- `src/main/services/tools/permission-policy.ts`
- `src/main/services/tools/tool-execution.service.ts`
- `src/main/ipc/handlers/*` 中与审批事件相关链路
- renderer 侧审批弹窗/任务面板

#### 核心设计

新增统一审批状态：

- `pending_approval`
- `approved`
- `rejected`
- `expired`

新增审批对象最小结构：

- `taskId`
- `runId`
- `toolCallId`
- `toolName`
- `arguments`
- `riskLevel`
- `reason`
- `requestedAt`
- `resolvedAt`
- `decision`

#### 验收标准

- `bash` / `file_write` / 主动浏览器工具在 `confirm` 模式下不会自动执行
- 用户批准前任务停在 `pending_approval`
- 用户批准后任务从原位置继续
- 用户拒绝后任务输出明确错误并进入终态
- 全链路审计可追踪

#### 业务收益

- 从“危险但可用”提升为“危险但可控”
- 是企业级使用与真实托付的前置条件

---

### P0-2 任务卡与验收卡产品化

#### 目标

降低模糊任务导致的失败率，把任务输入结构化。

#### 核心做法

所有中高复杂度任务在进入执行前，统一生成或要求补齐以下字段：

- 目标
- 输入文件
- 允许修改范围
- 禁止修改范围
- 执行步骤
- 验收标准
- 输出归档

#### 推荐模板

```md
### 任务ID：<task-id>
- 目标：
- 输入：
  - 文件：
  - 日志：
- 允许修改：
- 禁止修改：
- 执行步骤：
- 验收标准：
- 输出归档：
```

#### 需要改造的链路

- `SmartRouter` 前置任务分析
- `Workforce` 分解前的任务结构化提示
- renderer 发任务入口
- 可能需要一个轻量 task brief builder

#### 验收标准

- 中高复杂度任务自动或半自动形成任务卡
- Agent 输出带任务 ID 与验收项对齐关系
- 越界修改率显著下降

---

### P0-3 建立 Agent 任务能力边界矩阵

#### 目标

明确“谁适合做什么任务”，降低误路由与错误托付。

#### 需要输出的矩阵内容

按 Agent / Category 明确：

- 默认角色
- 默认策略
- 允许工具
- 禁止工具
- 适合任务类型
- 不适合任务类型
- 风险等级
- 推荐模型关键词

#### 参考现有定义

- `src/shared/agent-definitions.ts`
- `src/main/services/delegate/tool-allowlist.ts`

#### 验收标准

- 每个 Agent/Category 都有明确任务边界
- 路由层能够使用该矩阵辅助决策
- UI 可解释“为什么这个任务被分给它”

---

## 6.2 P1：把系统从“可托付”提升到“可稳定恢复”

### P1-1 持久化中间执行过程

#### 目标

让长任务在刷新、崩溃、切会话后仍能恢复执行现场，而不仅是恢复到最终结果。

#### 当前短板

- 流式输出主要在 renderer 内存
- tool call 过程不是完整持久化事件流
- 会话切换会压缩非当前 session 的 streaming state

#### 改造建议

新增持久化事件流表或附属结构，至少记录：

- `message-stream-started`
- `tool-call-requested`
- `tool-call-approved/rejected`
- `tool-call-completed`
- `llm-response-chunked`
- `checkpoint-saved`
- `run-paused`
- `run-resumed`

#### 推荐落点

- main 侧：message / task / run 服务
- renderer 侧：streaming store 改为从持久层回放

#### 验收标准

- 刷新后可恢复最近一次任务的关键中间态
- 审计可查看完整执行过程
- 任务接管者可快速理解卡点

---

### P1-2 统一重试与恢复策略

#### 目标

把当前 workflow retry、LLM retry、tool error retry 的口径统一起来，避免不同层各自判断导致行为漂移。

#### 当前问题

- Workflow 有较完整的分类和 backoff
- LLM 层也有独立的 retry helper
- 不同模块对可重试/不可重试的判断可能不一致

#### 改造建议

统一出一份共享契约：

- 错误分类
- 是否可重试
- 最大次数
- 退避策略
- 是否需要人工接管
- 是否允许切换模型/Agent/Category

#### 验收标准

- 相同错误在不同链路下得到一致判断
- 日志里能看到统一分类与下一步动作
- 人工接管条件清晰可解释

---

### P1-3 增强恢复语义上下文

#### 目标

把“恢复到能继续执行”提升为“恢复到能理解当前决策状态再继续执行”。

#### 需要新增的恢复摘要内容

- 当前任务目标
- 已完成事项
- 未完成事项
- 最近关键决策
- 禁止回退的约束
- 最近失败原因
- 建议下一步动作

#### 验收标准

- 长任务恢复后，模型不再轻易推翻已完成工作
- 恢复 prompt 的有效性提升
- 重复劳动和无效回滚显著减少

---

## 6.3 P2：把系统从“能恢复”提升到“更会分配、更会完成”

### P2-1 升级 Router 为“规则 + 语义”双模路由

#### 目标

降低复杂任务误路由率，提高“第一次就选对执行模式”的概率。

#### 当前问题

单纯正则规则在以下情况容易失真：

- 用户换一种说法描述同类任务
- 一个任务同时包含设计、编码、测试、发包多种意图
- 任务虽然看起来像简单任务，但实际涉及高风险动作

#### 改造建议

在现有 Router 基础上增加轻量语义判断层：

- 任务复杂度评分
- 风险评分
- 信息充分度评分
- 是否需要人工审批评分
- 是否适合 Workforce 的评分

#### 验收标准

- 路由解释中包含评分依据
- 同类任务的路由一致性更高
- “走错模式”导致的返工率下降

---

### P2-2 引入“任务类型模板库”

#### 目标

通过任务模板把常见任务标准化，提高成功率和结果可比性。

#### 建议优先支持的模板

- Bug 修复任务
- 小功能开发任务
- 重构任务
- 文档任务
- 测试补齐任务
- 发版验证任务
- 浏览器自动化任务

#### 模板至少包含

- 默认策略（direct / delegate / workforce）
- 推荐 Agent / Category
- 建议工具作用域
- 风险级别
- 推荐验收项

#### 验收标准

- 用户发任务时可套用模板
- 模板任务完成率高于自由输入任务
- 模板输出更稳定、更便于验收

---

### P2-3 建立统一的“卡点诊断面板”

#### 目标

让用户可以快速知道“它为什么没做完”。

#### 面板建议展示

- 当前阶段
- 当前子任务
- 当前阻塞类型
- 最近工具调用
- 最近失败分类
- 是否等待审批
- 是否建议人工接管

#### 验收标准

- 用户无需翻多处日志即可理解卡点
- 人工接管时间明显缩短

---

## 6.4 P3：把系统从“研发辅助”提升到“平台级自治执行”

### P3-1 建立任务能力基准集

#### 目标

用固定任务集持续评估 Agent 真实完成能力。

#### 建议建立的基准集维度

- 简单代码任务
- 跨模块实现任务
- 只读分析任务
- 高风险审批任务
- 浏览器自动化任务
- 恢复/续跑任务
- 发布验收任务

#### 每个基准任务记录

- 输入描述
- 期望路由
- 允许工具
- 期望输出
- 是否需要审批
- 是否通过
- 失败原因

#### 验收标准

- 每周/每版本可以跑一次基准集
- 能用数据判断优化是否有效

---

### P3-2 建立任务级 KPI 仪表盘

#### 建议指标

- 任务完成率
- 一次通过率
- 平均重试次数
- 平均人工接管次数
- 危险动作审批命中率
- 越界修改率
- 中途丢失上下文率
- 跨会话恢复成功率

#### 验收标准

- 每个版本都能看到趋势变化
- 能快速定位是 Router、Delegate、Workforce、Tool、UI 哪一层退化

---

### P3-3 跨平台发布闭环

#### 目标

补齐 Windows 发布与启动验证，使系统具备平台级交付可信度。

#### 当前阻塞

- Windows build 仍有失败记录
- Windows boot 在当前证据中仍是 blocked

#### 建议动作

- 建立原生 Windows CI / Runner
- 固化 `build -> install -> boot -> core flow -> archive evidence` 的门禁流程
- 将发版门禁与 Agent 平台能力门禁分层管理

#### 验收标准

- Windows/macOS 核心门禁全部可回归
- 发布签署不再依赖人工口头判断

---

## 7. 推荐实施顺序

建议按下面顺序推进，而不是并行全部开工：

1. **P0-1 真审批闸门**
2. **P0-2 任务卡产品化**
3. **P0-3 能力边界矩阵**
4. **P1-1 中间态持久化**
5. **P1-2 重试恢复统一**
6. **P1-3 恢复语义增强**
7. **P2-1 Router 双模升级**
8. **P2-2 任务模板库**
9. **P2-3 卡点诊断面板**
10. **P3-1 / P3-2 / P3-3 平台化建设**

理由如下：

- 没有审批闸门，不能托付
- 没有任务卡，成功率难提高
- 没有边界矩阵，路由始终会漂
- 没有中间态持久化，长任务很难稳定
- 没有指标体系，后续优化很难证明有效

里程碑结论：

**系统达到“可承接中长链路研发任务”的水平。**

- 基准集稳定提升
- KPI 仪表盘可持续跟踪

里程碑结论：

**系统达到“平台级自治执行系统”的雏形。**

## 12. 最终结论

从任务承接视角看，**CodeAll 现在已经不是“演示型 Agent 项目”，而是一个具备真实执行链路的 Beta 级任务执行系统**。它已经能完成不少真实研发任务，但距离“高风险场景下稳定自治交付”还差最后一段工程化路程。

这段路不是补更多 Agent 名字，也不是继续堆更多工具，而是要优先补齐：

- **审批闸门**
- **任务结构化**
- **中间态持久化**
- **恢复语义化**
- **能力边界与指标体系**

把这几件事补齐之后，CodeAll 才会从“会做事”真正升级成“值得托付任务”。
