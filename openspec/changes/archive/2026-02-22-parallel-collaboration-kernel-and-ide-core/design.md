## Context

CodeAll 目前已具备基础能力：
- 后端已有多 agent 工作流骨架（任务分解、DAG 执行、delegate 调用、基础重试与事件桥接）。
- 前端已有多面板工作台、流程图、产物预览、内嵌浏览器与 AI 浏览器工具。

但系统仍存在两类结构性缺口：
1. **协同内核不完整**：任务分配、共享上下文、资源调度、结果整合与恢复语义尚未形成统一治理内核。
2. **IDE 核心闭环不足**：代码编辑、终端/后台任务中心、运行日志联动与可观测闭环不完整。

本设计采用并行方案 C：在同一主变更下并行推进“协同内核”和“IDE 核心能力”，并通过统一事件合同与验收指标进行集成。

约束：
- 保持与《项目规划.md》目标一致；
- 兼容现有 Electron 主/渲染进程架构与 Prisma 数据层；
- 优先复用现有模块，不做无关大重写。

## Goals / Non-Goals

**Goals:**
- 定义可执行的“协同内核”架构：分解、调度、上下文共享、结果整合、恢复重试、运行观测。
- 定义“IDE 核心闭环”架构：编辑、执行、追踪、审阅、浏览器自动化与设置治理。
- 建立异构模型协作下的路由与资源分配规则（agent/category/model/provider）。
- 建立统一的任务/运行事件合同，支撑跨面板追踪与问题定位。
- 采用分阶段迁移，保证现有功能可持续运行。

**Non-Goals:**
- 不在本设计阶段引入与目标无关的新产品线（如插件市场、云协作平台化改造）。
- 不追求一次性重写全部 workforce/delegate/router 代码。
- 不在本设计阶段扩展全新外部基础设施（如分布式队列集群）。

## Decisions

### Decision 1: 协同内核采用“四层模型”而非单体 workflow 扩展
- 选择：将协同能力拆分为 `Planner/Graph`、`Scheduler/Allocator`、`Context Bus`、`Integrator` 四层。
- 原因：当前逻辑集中在单引擎内，扩展后易耦合；分层后便于并行开发与独立测试。
- 备选：继续在现有 `workforce-engine` 增量堆叠。
  - 放弃原因：复杂度持续上升，职责边界不清晰，难以做稳定性治理。

### Decision 2: 工作流图与依赖关系采用“分阶段持久化（metadata→结构化）”
- 选择：现阶段在 Task/Run 上通过标准化 `metadata.graph / metadata.timeline / metadata.integration / metadata.retryState` 承载工作流结构与观测数据，并保留后续迁移到显式结构化表的演进路径。
- 原因：在不引入高风险迁移的前提下，先保障恢复、追踪、重放、统计与跨面板关联可用；随后再按容量与查询需求升级为独立结构化存储。
- 备选：一次性完成全量结构化持久化。
  - 放弃原因：改造面过大、迁移与回滚成本高，不利于当前并行交付节奏。

### Decision 3: 引入共享上下文总线（Context Bus）
- 选择：定义标准上下文对象（facts/decisions/constraints/artifacts/dependency outputs），按角色注入。
- 原因：避免仅靠文本拼接与截断依赖，提升跨 agent 协作一致性。
- 备选：每个子任务只读父 prompt 与依赖摘要。
  - 放弃原因：信息丢失严重，难做稳定的多 agent 结果协同。

### Decision 4: 资源调度采用模型感知并发键（concurrency key）
- 选择：调度键支持 agent/category/provider/model 维度，结合默认并发策略与配额。
- 原因：异构模型协作下，固定并发无法兼顾效率与稳定。
- 备选：固定全局并发上限。
  - 放弃原因：易出现热点拥塞与资源利用不均。

### Decision 5: 结果输出增加“整合阶段（Integrator）”
- 选择：在 workflow 完成前执行整合步骤，对子结果做一致性检查、冲突消解与结构化汇总。
- 原因：直接拼接结果不可控，无法保证质量与可追踪性。
- 备选：保持 join 文本式汇总。
  - 放弃原因：无法满足规划文档对协同质量与可维护性的要求。

### Decision 6: IDE 核心按“工作台能力面板化”推进
- 选择：在现有多面板布局上，补齐编辑器、终端/后台任务中心、日志追踪联动；不推翻现有 UI 架构。
- 原因：当前布局基础已在，增量扩展成本最低。
- 备选：重做整套前端框架。
  - 放弃原因：风险高、周期长、与当前产线不匹配。

### Decision 7: 浏览器自动化能力保持“内嵌服务 + 明确权限边界”
- 选择：延续内嵌 BrowserView + AI browser tools 的架构，并补充策略治理与可观测记录。
- 原因：现有能力可用，重点是工作流集成与审计完善。
- 备选：引入全新外部浏览器编排系统。
  - 放弃原因：集成成本高，增加维护面。

### Decision 8: 设置治理采用“统一绑定与策略校验”
- 选择：模型/Provider/API Key/Agent 绑定统一走配置治理层，路由执行前做一致性校验。
- 原因：减少“配置可见但运行时失效”的问题。
- 备选：维持分散配置检查。
  - 放弃原因：排障成本高，行为不可预测。

## Risks / Trade-offs

- [协同内核改造跨模块范围大] → 采用分阶段交付与兼容适配层，先接入新调度与整合，再替换旧路径。
- [数据结构调整引入迁移风险] → 设计可回滚迁移脚本，保留旧字段读取窗口，逐步切换写路径。
- [并行推进导致前后端契约漂移] → 建立 IPC 合同校验清单与回归检查，保证声明/注册/预加载/前端消费一致。
- [模型感知调度增加策略复杂度] → 先实现最小可用规则集（固定优先级 + 配额），后续再做自适应优化。
- [新增面板与日志联动影响 UI 性能] → 引入事件节流与增量渲染，限制高频日志默认渲染窗口。
- [结果整合阶段增加时延] → 对整合器设置超时与降级策略，确保可用性优先。

## Migration Plan

1. **阶段 1：协同内核骨架落地**
   - 建立四层职责边界与接口；
   - 将现有 workflow 执行接入新调度/整合入口（保持旧路径兼容）。

2. **阶段 2：数据与观测能力升级**
   - 增加工作流结构化持久化；
   - 建立任务/运行/工作流统一关联与查询接口。

3. **阶段 3：IDE 核心闭环补齐**
   - 接入编辑器写回链路；
   - 打通终端/后台任务中心与日志追踪联动；
   - 强化任务-产物-运行跨面板跳转。

4. **阶段 4：浏览器与设置治理收敛**
   - 补齐浏览器自动化审计与策略控制；
   - 统一模型绑定与安全校验逻辑。

5. **阶段 5：验收与回归**
   - 依据规格验收多 agent 协同、IDE 核心、稳定性与性能；
   - 对关键路径执行回归测试并完成文档更新。

回滚策略：
- 以阶段为单位回滚，保留旧执行路径；
- 数据层采用向后兼容读取，必要时恢复到旧写入路径。

## Open Questions

- 共享上下文总线的持久化粒度应到“任务级”还是“工作流级”，或二者并存？
- 整合阶段采用专门 Integrator agent 还是规则化整合器 + 可选 agent 审阅？
- 模型并发配额默认策略应按 provider 还是按 model 精细化配置？
- IDE 编辑器首版是否仅支持文本/代码，二进制与富媒体编辑是否延后？
- 运行日志默认保留窗口和归档策略如何设定，才能兼顾性能与审计？
