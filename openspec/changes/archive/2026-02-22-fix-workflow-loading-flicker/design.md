## Context

当前流程图交互在“节点点击 → 工作流详情加载 → 详情面板渲染”链路中存在两个核心问题：
1. 详情请求与 UI 状态绑定不稳定，快速连续点击或慢响应时容易出现旧请求覆盖新选择的竞态；
2. 渲染层在加载阶段发生视图清空与重复挂载，导致“加载工作流…”长期可见并出现明显闪烁。

该链路横跨 Renderer 组件、状态管理、IPC 请求与事件桥接，属于跨模块一致性问题；修复需要统一状态机语义与请求归属策略，而不仅是单点样式调整。

## Goals / Non-Goals

**Goals:**
- 建立稳定、可收敛的工作流详情加载状态机（`idle/loading/ready/error`）。
- 消除异步请求竞态导致的“悬挂加载”与错误数据回退。
- 降低加载阶段可见闪烁，保证流程图和详情区域在切换期间视觉连续。
- 为高频点击与延迟响应场景提供可回归测试。

**Non-Goals:**
- 不改动工作流编排算法与后端任务执行语义。
- 不引入新的前端状态管理库或全局架构重写。
- 不扩展流程图功能（如新增节点类型、编辑能力）。

## Decisions

1. **采用“请求代次（request token）+ 归属校验”机制约束异步响应生效范围**
   - 决策：每次用户选择工作流时生成新的请求代次，仅当响应代次仍是当前活动代次时才提交状态。
   - 原因：可在不改变 IPC 协议主体的前提下，最小侵入地解决旧响应覆盖新状态问题。
   - 备选方案：
     - 直接取消前一次请求（AbortController）——对跨进程 IPC/事件流的覆盖有限，且仍需处理已返回响应。
     - 全量串行化请求——会增加交互等待，不利于快速切换体验。

2. **将 UI 渲染切换为“稳定容器 + 内容渐进更新”而非“加载时清空后重建”**
   - 决策：在加载新详情期间保留上一次有效内容或固定骨架占位，仅替换内容区状态标识，避免容器级卸载。
   - 原因：闪烁主要来自 DOM 树大范围重建；保持容器稳定可直接降低视觉抖动。
   - 备选方案：
     - 增加加载遮罩覆盖全屏——实现简单，但会掩盖竞态问题且交互阻断更强。
     - 纯 CSS 动画优化——无法解决状态反复切换导致的根因。

3. **统一详情加载状态来源，避免组件内局部 loading 与全局 loading 相互打架**
   - 决策：以单一来源状态（store 或会话级状态）驱动详情面板与流程图选中态，减少重复 `setLoading` 入口。
   - 原因：当前闪烁由多处状态更新顺序不一致放大；单一真相源可减少重绘震荡。
   - 备选方案：
     - 保留多状态源并增加防抖——只能缓解，不保证一致性。

4. **增加回归测试覆盖竞态和渲染稳定性**
   - 决策：补充“快速连续点击 + 慢响应返回顺序颠倒”测试，断言最终展示与最后一次选择一致，且加载状态可收敛。
   - 原因：该问题高度依赖时序，必须通过可重复测试防止回归。
   - 备选方案：
     - 仅手工验证——难以覆盖边界时序，回归风险高。

## Risks / Trade-offs

- **[风险]** 保留上一次有效视图可能在短时间显示“旧内容” → **缓解**：在 UI 明确标识“正在加载新工作流”，并在新数据就绪后原子切换。
- **[风险]** 请求代次策略若实现分散，仍可能遗漏更新路径 → **缓解**：将代次校验封装在统一加载入口，禁止旁路提交。
- **[风险]** 测试中模拟时序可能不稳定 → **缓解**：使用可控 mock 时钟/延迟与固定响应顺序，减少非确定性。
- **[权衡]** 为稳定渲染引入额外状态字段，增加少量代码复杂度 → **收益**：换取交互一致性与可维护性。
