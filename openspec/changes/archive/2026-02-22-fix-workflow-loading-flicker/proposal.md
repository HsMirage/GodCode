## Why

在开发服务器测试中，用户点击流程图后界面长时间停留在“加载工作流…”，并伴随明显闪烁，导致无法稳定查看工作流详情和节点执行状态。该问题直接影响核心排障与协作体验，需要优先修复以恢复流程可观测性与交互连续性。

## What Changes

- 规范流程图点击后的数据加载生命周期：区分 `idle/loading/ready/error`，避免重复触发导致的状态抖动。
- 引入请求竞争保护（例如基于请求标识的响应归属判断），确保旧请求返回不会覆盖新选择的工作流状态。
- 调整前端渲染切换策略：在新数据就绪前保持稳定占位或上一次有效视图，消除空白-重绘造成的闪烁。
- 为“点击节点/工作流 → 加载详情 → 渲染完成”链路补充可验证测试与观测点，覆盖快速连续点击与慢响应场景。

## Capabilities

### New Capabilities
- `workflow-ui-loading-stability`: 约束流程图相关 UI 在详情加载过程中的稳定状态流转与防闪烁行为。

### Modified Capabilities
- `workflow-observability-and-recovery`: 明确 UI 获取工作流连续状态时的可预期返回与状态一致性要求，避免“持续加载”悬挂。
- `ide-core-edit-exec-workbench`: 强化从流程节点交互到日志/产物视图切换的平滑性要求，禁止因异步竞争导致的可见闪烁。

## Impact

- 受影响系统：工作流详情加载链路、流程图节点交互、执行轨迹联动视图。
- 受影响代码（预期）：Renderer 中流程图与详情面板相关组件、状态管理逻辑、Main/IPC 的工作流详情查询与事件桥接。
- 测试影响：需要新增/调整前端交互测试与集成测试，验证加载状态收敛、竞态覆盖与无闪烁渲染行为。
- 外部 API/依赖：不引入新外部依赖，主要为现有状态流转与请求协同逻辑优化。
